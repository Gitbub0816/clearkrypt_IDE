# Target Mappings

> This document supports the ClearKrypt Constitution. If it conflicts with
> `/constitution`, the constitution wins. It records the exact, honest mapping
> from ClearKrypt declarations to Swift, Kotlin, and React/TypeScript output
> for the current compiler version. Emitters implement this document; changes
> to emitters must update this document.

## Generated output layout

Per Constitution Document 6 §11:

```text
generated/
  swift/
    ClearKrypt.swift          # shared support typealiases
    app/main/Greeting.swift   # one file per declared type
    app/main/Functions.swift  # free functions for module app.main
  kotlin/
    clearkrypt/ClearKrypt.kt  # shared support typealiases (package clearkrypt)
    app/main/Greeting.kt
    app/main/Functions.kt
  react/
    clearkrypt.ts             # shared support type aliases
    app/main.ts               # one file per module
```

Every generated file begins with a header naming the generator, compiler
version, source file, and module:

```text
Generated by ClearKrypt <version> from src/main.ck (module app.main).
Do not edit. Changes will be overwritten.
```

rendered in the target's comment syntax. Generated files are deterministic:
identical input produces byte-identical output.

## Primitive type mappings

| ClearKrypt | Swift        | Kotlin                 | TypeScript   | Notes |
| ---------- | ------------ | ---------------------- | ------------ | ----- |
| String     | `String`     | `String`               | `string`     | |
| Int        | `Int`        | `Int`                  | `number`     | TS `number` is a 64-bit float; integer values beyond 2^53 lose precision. Documented, honest limitation. |
| Float      | `Double`     | `Double`               | `number`     | ClearKrypt `Float` means a 64-bit float on every target. |
| Decimal    | `Decimal`    | `java.math.BigDecimal` | `Decimal` (alias of `string`) | TypeScript has no decimal type; the alias documents intent. |
| Bool       | `Bool`       | `Boolean`              | `boolean`    | |
| Date       | `Date`       | `java.time.LocalDate`  | `string`     | Calendar date. Swift uses `Date` at midnight UTC (OpenAPI convention). TS uses ISO-8601 `YYYY-MM-DD`. |
| DateTime   | `Date`       | `java.time.Instant`    | `string`     | An instant in time. TS uses ISO-8601. |
| ID         | `ID` (alias of `String`)    | `ID` (alias)    | `ID` (alias of `string`)    | Alias defined once in the support file. |
| Email      | `Email` (alias of `String`) | `Email` (alias) | `Email` (alias of `string`) | |
| URL        | `URL` (Foundation)          | `URL` (alias of `String`) | `URL` (alias of `string`) | Swift uses the native Foundation type. |
| Data       | `Data`       | `ByteArray`            | `Uint8Array` | |
| Void       | (omitted)    | `Unit`                 | `void`       | Return positions only. |
| Never      | `Never`      | `Nothing`              | `never`      | Return positions only. |

Semantic string primitives (`ID`, `Email`, `URL` on non-Swift targets) are
aliases of the target string type, defined once in the per-target support
file. In the type checker they are mutually assignable with `String` in this
version; stricter branding is a possible future feature and would be a
breaking change announced in docs.

## Collections

| ClearKrypt   | Swift       | Kotlin         | TypeScript              |
| ------------ | ----------- | -------------- | ----------------------- |
| List&lt;T&gt;      | `[T]`       | `List<T>`      | `readonly T[]`          |
| Map&lt;K, V&gt;    | `[K: V]`    | `Map<K, V>`    | `ReadonlyMap<K, V>`     |
| Set&lt;T&gt;       | `Set<T>`    | `Set<T>`       | `ReadonlySet<T>`        |
| T?           | `T?`        | `T?`           | `T | null`              |

Optionality means the value may be absent. On TypeScript, absence is `null`
(not `undefined`) so that round-tripping through JSON stays explicit.

## Models

ClearKrypt models are immutable value data.

- **Swift**: `public struct <Name>: Hashable` with `public let` fields and an
  explicit `public init` (memberwise inits are internal-only in Swift).
  Field defaults become init parameter defaults.
- **Kotlin**: `data class <Name>(...)` with `val` parameters. Field defaults
  become constructor parameter defaults.
- **TypeScript**: `export interface <Name>` with `readonly` fields. When a
  model has default values, the emitter also generates a
  `create<Name>(init)` factory function that applies them, because interfaces
  cannot express defaults.

`Hashable` is emitted on Swift structs and enums so `Set` and `Map` keys work
across targets without extra ceremony.

## Enums

Simple enums (no associated values):

- **Swift**: `public enum <Name>: String, Hashable` with one `case` per case.
- **Kotlin**: `enum class <Name>` with SCREAMING_SNAKE-free, original-case
  entries adapted to Kotlin conventions (`pending` -> `Pending`).
- **TypeScript**: string literal union: `export type X = 'pending' | 'shipped'`.

Enums with associated values:

- **Swift**: native enum with associated values:
  `case cancelled(reason: String)`.
- **Kotlin**: `sealed interface <Name>` with `data object` for empty cases and
  `data class` for cases with values (`Cancelled(val reason: String)`).
- **TypeScript**: discriminated union with a `kind` discriminant:
  `{ kind: 'cancelled'; reason: string }`.

Case names remain traceable: `cancelled` stays `cancelled` (Swift, TS
discriminant) or becomes `Cancelled` (Kotlin type name), per target
convention. No arbitrary renaming.

## Errors

`error` declarations map like associated enums with error semantics:

- **Swift**: `public enum <Name>: Error, Hashable` with associated values.
- **Kotlin**: `sealed class <Name> : Exception()` with `data object` /
  `data class` cases.
- **TypeScript**: discriminated union type (same shape as enums). Error
  propagation strategy (Result types vs exceptions) is defined in
  `10-error-async-effect-model.md` and lands with a later milestone.

## Functions

Pure functions map one-to-one:

- **Swift**: `public func name(param: Type) -> Type`. Call sites include
  Swift argument labels matching ClearKrypt parameter names.
- **Kotlin**: top-level `fun` in `Functions.kt` for the module's package.
- **TypeScript**: `export function` in the module file.

Async and typed-throws signatures:

- **Swift**: `async` + plain `throws`, with a `/// - Throws: <ErrorType>` doc
  comment. (Swift typed throws may be adopted later as a target profile.)
- **Kotlin**: `suspend fun`, with the error type documented in KDoc. Kotlin
  has no checked exceptions; this is a documented target difference.
- **TypeScript**: `async function ... : Promise<T>`, error type documented in
  TSDoc.

Model construction (`User(id: x, name: y)`) emits:

- **Swift**: `User(id: x, name: y)`
- **Kotlin**: `User(id = x, name = y)`
- **TypeScript**: `{ id: x, name: y }` object literal (checked against the
  interface).

Nested (local) functions — `fn` as a statement inside any block — reuse the
exact same signature and call rendering as top-level functions, indented in
place:

- **Swift**: a local `func`, which captures the enclosing scope natively.
- **Kotlin**: a local `fun`, which captures the enclosing scope natively.
- **TypeScript**: a nested `function`, which captures the enclosing scope
  natively (a closure).

No target needs special machinery for the capture itself; each of the three
languages already gives nested functions access to their enclosing scope.
Self-recursive and mutually-referential calls between local functions resolve
through the same call-resolution path as top-level calls (same module, so no
cross-module import is added).

## Expression and control-flow mappings

| ClearKrypt | Swift | Kotlin | TypeScript |
| ---------- | ----- | ------ | ---------- |
| `"a\(x)b"` | `"a\(x)b"` | `"a${x}b"` (with `$` escaped in literals) | `` `a${x}b` `` |
| `a ?? b` | `a ?? b` | `a ?: b` | `a ?? b` (parenthesized when mixing with `&&`/`\|\|`) |
| `a?.b` | `a?.b` | `a?.b` | `(a?.b ?? null)` — normalized so absence stays `null`, except directly before `??` |
| `if let x = v { }` | `if let x = v { }` | `val x = v; if (x != null) { }` (smart cast) | `const x = v; if (x !== null) { }` (narrowing) |
| `Status.pending` | `Status.pending` | `Status.Pending` | `'pending'` (simple) / `{ kind: 'pending' }` (associated) |
| `E.server(message: m)` | `E.server(message: m)` | `E.Server(message = m)` | `{ kind: 'server', message: m }` |
| `match` | `switch` expression (Swift 5.9+) | `when` expression (`when (val matched = ...)` only when payload arms bind) | exhaustive `switch` (on the value or `.kind`), IIFE in let position, plus an unreachable-guard `throw` |
| `throw e` | `throw e` | `throw e` (errors extend `Exception`) | `throw e` (plain tagged object; helper guards are a later milestone) |
| `try f(x)` | `try f(x)` | `f(x)` (JVM propagation) | `f(x)` (JS propagation) |

Match exhaustiveness is proven by the checker (CK0014), so the generated
switches never rely on target-side exhaustiveness analysis — but they keep
it where the target offers it (Kotlin `when` over sealed types).

## Not yet emitted (honest limitations)

Per the target honesty law, the following parse and type-check but are
**not** emitted yet. The compiler reports diagnostic `CK0004` (warning) for
each such declaration at lowering time, rather than silently dropping them
(missing native implementations for selected targets are `CK0005` errors):

- screens and components (Roadmap Milestone 7: SwiftUI / Compose / React)
- routes (Milestone 7)
- native function blocks (Milestone 10: interop build inclusion)
- capabilities and `requires` clauses (parsed and recorded; permission
  emission is a later milestone)
- serialization support (Codable / kotlinx.serialization / JSON schema) is
  deferred; models are plain values today

## Known target limitations

- **Kotlin `Data` fields**: `Data` maps to `ByteArray`, and array properties
  do not participate in `data class` structural `equals`/`hashCode` the way
  value collections do. This is a documented Kotlin platform behavior, not a
  ClearKrypt bug; a value-semantics wrapper is a candidate improvement.
- **Async call sites**: `async` functions parse, check, and emit with the
  right signatures, but calling one requires `await` support, which lands
  with the async milestone. The checker does not yet reject cross-calls.
- **IR expression origins**: declarations, fields, and params carry source
  origins; individual expressions do not yet. Emitter diagnostics for
  malformed expressions cite the enclosing declaration's span. Tightening
  this is tracked for a later IR revision.

## Stability rules

- Output is deterministic and stable across runs (snapshot-tested).
- Declaration order in generated files follows source order.
- Generated files never mix with handwritten native code (`native/` vs
  `generated/`).
